# Pull Request Review 入门指南

Review 是团队协作里非常重要的一环。它不是挑错大会，而是让修改更清楚、更安全、更容易维护的过程。

## 适合人群

- 第一次发 Pull Request 的新手。
- 第一次帮别人看 PR 的同学。
- 想知道 Review 应该关注什么的人。

## 发 PR 前自查

提交 PR 之前，先确认：

- 本次修改范围是否清楚。
- 文件名、目录位置是否合理。
- README 或入口导航是否更新。
- 文档链接是否能打开。
- 命令、代码块、路径是否写对。
- 是否包含密码、Token、私钥、临时文件。
- 提交信息是否能说明改动目的。

如果是文档项目，建议运行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\check-docs.ps1
```

## PR 描述模板

```markdown
## 修改内容

- 新增 xxx 文档
- 更新 README 导航

## 验证

- 已运行文档检查脚本
- 已手动检查新增链接

## 备注

- 暂无
```

PR 描述不用很长，但要让 Reviewer 知道你希望他看什么。

## Reviewer 看什么

Review 时优先看：

1. 是否解决了 PR 描述里的问题。
2. 是否有无关改动。
3. 是否破坏已有链接或入口。
4. 是否有安全风险。
5. 是否对新手足够清楚。

文档 Review 不只是看错别字。更重要的是看读者能不能顺着文档完成操作。

## 评论怎么写

好的评论应该具体、可执行。

不太好：

```text
这里不行。
```

更好：

```text
这里建议补一行“如何验证成功”，否则新手不知道 SSH 密钥是否配置完成。
```

如果只是个人偏好，可以标明：

```text
非阻塞建议：这里可以改成表格，读起来更快。
```

## 收到 Review 后怎么处理

建议顺序：

1. 先读完所有评论。
2. 把必须修改的问题先处理。
3. 对建议类评论判断是否采纳。
4. 修改后继续提交到同一个分支。
5. 回复说明已处理或解释原因。

不要把每条评论都当成否定。Review 是协作记录，也是学习材料。

## 常见误区

| 误区 | 建议 |
| :--- | :--- |
| PR 越大越厉害 | 小而清晰的 PR 更容易合并。 |
| Review 只看代码 | 文档、命名、入口、风险都要看。 |
| 收到评论就重新开 PR | 通常继续推送到同一分支即可。 |
| 不同意就沉默 | 说明原因，保留讨论记录。 |
