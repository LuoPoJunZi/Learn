从“云端轻量修改”正式过渡到了**“本地专业开发”**。这也是全世界绝大多数高级程序员每天都在使用的标准工作流。

既然你已经新建了 `Data` 文件夹并用 VS Code 打开了它，下面我手把手教你如何把项目拉下来，以及以后如何丝滑地修改和提交代码。

### 🚀 第一步：把项目“拉”到你的本地电脑 (Clone)

现在你的 `GithubData` 文件夹还是空的，我们需要把云端的代码“克隆”下来。

1. 在 VS Code 中，点击顶部菜单栏的 **Terminal (终端)** -> **New Terminal (新建终端)**。
2. 在下方弹出的终端窗口里，输入以下命令并回车：
   ```bash
   git clone https://github.com/XXXXXX（用户名）/XXX（仓库名称）.git
   ```
3. 等待几秒钟，你的左侧资源管理器里就会神奇地多出一个叫 `sing-box-luopo` 的文件夹！
4. **⚠️ 极其重要的一步**：在 VS Code 里点击 **File (文件)** -> **Open Folder (打开文件夹)**，然后选中刚刚生成的那个 `sing-box-luopo` 文件夹并打开。
   *(只有当你打开的是项目根目录时，VS Code 才能正确识别代码库！)*

---

### 💻 第二步：日常的修改与提交 (Commit & Push)

以后你的日常开发就会变成极其舒适的“可视化操作”：

1. **修改代码**：直接在左侧点开文件（比如 `src/init.sh`），修改内容，按 `Ctrl + S` 保存。
2. **查看更改**：点击 VS Code 最左侧边栏的 **源代码管理** 图标（长得像一个网络分支 ᛦ，通常会有个蓝色小数字提示你改了几个文件）。
3. **暂存 (Stage)**：在“更改 (Changes)”列表里，鼠标悬停在你修改的文件上，点击右侧的 **`+` 号**（这相当于运行 `git add`）。文件会跑到“暂存的更改”里。
4. **提交 (Commit)**：在上面的输入框里写一句简短的说明（比如：`更新版本号到 v1.3`），然后点击蓝色的 **Commit (提交)** 按钮。
5. **推送云端 (Push)**：提交完之后，原来蓝色的按钮会变成 **Sync Changes (同步更改)**，点一下它！代码就瞬间飞到 GitHub 云端了，流水线也会自动开始打包！

### ⚖️ 本地 VS Code vs. 网页端 GitHub.dev 到底有啥区别？

你之前用的网页端（按 `.` 键弹出的那个）确实很方便，但它只是个“阉割版”的阅读器。本地 VS Code 才是真正的“重型武器”。

| 功能维度 | 💻 你的本地 VS Code (本地环境) | 🌐 GitHub.dev (网页端) |
| :--- | :--- | :--- |
| **真实终端 (Terminal)** | **✅ 有！** 这是最大的区别。你可以在下面直接敲 Linux/Git 命令，甚至能安装插件在本地模拟运行 Bash 脚本测试。 | **❌ 没有。** 网页端只是个纯文本编辑器，无法运行任何代码。 |
| **插件生态** | **✅ 完整。** 你可以安装各种强大的代码高亮、格式化、纠错神仙插件（比如 ShellCheck，能帮你自动检查脚本语法错误）。 | **⚠️ 受限。** 只能用一小部分支持 Web 的轻量插件。 |
| **网络要求** | **✅ 支持离线。** 在高铁上、没网的时候依然可以写代码、做 Commit 存档，有网了再 Push。 | **❌ 必须在线。** 断网就打不开，也无法保存。 |
| **全局搜索** | **✅ 极速且深入。** 可以在几千个文件里瞬间搜到某个变量，并且支持复杂的正则替换。 | **⚠️ 较慢。** 且有时搜索不全。 |
| **适用场景** | 适合深度开发、架构重构、批量修改、以及需要写代码超过 5 分钟的场景。 | 适合在外边临时借用别人电脑，紧急修一个错别字或改一个版本号。 |

**强烈建议你安装的一个 VS Code 插件：**
在 VS Code 左侧的扩展商店（四个方块那个图标）里，搜索并安装 **`shellcheck`**。这是一个专门针对 Bash 脚本的语法检查神器。以后你再写出像 `&;` 这样的语法错误，它会直接在代码下面画红波浪线提醒你，让你在推送到 GitHub 之前就把 Bug 扼杀在摇篮里！

你现在可以试着在本地修改一下 `README.md` 加个标点符号，然后用 VS Code 的可视化界面走一遍“暂存 -> 提交 -> 同步”的流程，感受一下丝滑的本地开发体验。遇到任何卡壳的地方随时问我！
