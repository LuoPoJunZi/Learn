# GitHub 真实场景指南

这篇文档把 GitHub 常见操作放到真实场景里讲。你不需要一次背完命令，先判断自己处在哪个场景，再按对应流程走。

## 场景一：维护自己的学习仓库

适合情况：

- 你主要修改 Markdown 笔记。
- 只有你一个人在维护仓库。
- 目标是把本地学习成果同步到 GitHub。

推荐流程：

```bash
git status
git add 文件名
git commit -m "docs: update study notes"
git push
```

关键习惯：

- 每次提交前先看 `git status`。
- 提交信息写清本次修改目的。
- 不要把密码、Token、临时文件提交上去。

如果你还不理解 `add`、`commit`、`push` 的区别，先看 [工作区、暂存区、本地仓库](docs/part2-git-cmds/2.1-core-concepts.md)。

## 场景二：把本地已有项目上传到 GitHub

适合情况：

- 电脑里已经有一个文件夹。
- 想把它变成 GitHub 仓库。
- 之前没有用 Git 管理。

推荐先读 [把本地已有文件夹推送到 GitHub](docs/part2-git-cmds/2.3-upload-local-project.md)。

操作前先检查：

- 文件夹里有没有 `.env`、密码、私钥、账号配置。
- 是否需要 `.gitignore`。
- README 是否写清项目用途。

新手常犯的错是：先 `git add .`，后来才发现把不该上传的文件也提交了。第一次上传时，宁可慢一点，一个文件一个文件确认。

## 场景三：给别人的开源项目贡献小修改

适合情况：

- 你没有对方仓库的写权限。
- 想修文档、补说明、改错别字、修一个小 bug。

推荐流程：

1. Fork 对方仓库。
2. Clone 自己账号下的 Fork。
3. 新建分支。
4. 修改文件。
5. 提交并推送到自己的 Fork。
6. 从 GitHub 页面发起 Pull Request。

PR 描述建议包含：

```text
## 修改内容

- 修正文档中的链接
- 补充安装步骤说明

## 验证

- 已本地打开 README 检查链接
- 已运行项目文档检查脚本
```

第一次贡献时，选择小范围修改更容易成功。不要一上来重构整个项目结构。

## 场景四：团队里用分支开发功能

适合情况：

- 多个人同时维护一个仓库。
- 修改需要别人 Review。
- `main` 分支需要保持稳定。

推荐流程：

```bash
git pull
git switch -c feature/add-docs
# 修改文件
git status
git add 文件名
git commit -m "docs: add usage guide"
git push -u origin feature/add-docs
```

然后在 GitHub 页面发起 Pull Request。

命名建议：

- `feature/...`：新功能或新增内容。
- `fix/...`：修复问题。
- `docs/...`：文档修改。
- `chore/...`：维护性修改。

## 场景五：收到 Review 意见

Review 不等于批评，它是团队一起把内容变好的过程。

处理顺序：

1. 先读完所有评论。
2. 判断哪些是必须修改，哪些是建议。
3. 小范围修改。
4. 继续提交到同一个分支。
5. 回复说明“已修改”或解释原因。

常用回复：

```text
已按建议补充示例，并更新 README 链接。
```

如果你不同意某条建议，不要直接忽略。说明你的考虑，团队协作靠的是可追溯的讨论。

## 场景六：合并前发现冲突

冲突常见于多人修改同一段内容。

处理前先做三件事：

1. 不要慌。
2. 不要随便删除整段文件。
3. 先读冲突上下文。

冲突标记长这样：

```text
<<<<<<< HEAD
当前分支内容
=======
要合并进来的内容
>>>>>>> other-branch
```

你需要人工决定最终保留什么，然后删除这些标记。详细说明看 [代码冲突怎么办](docs/part3-teamwork/3.3-merge-conflict.md)。

## 场景七：提交错了文件

如果还没提交：

```bash
git status
git restore --staged 文件名
```

如果已经提交但没推送，可以考虑新提交修正，或者在理解清楚后再撤销。

新手建议优先使用“再提交一次修正”的方式，少用历史改写命令。

## 场景八：什么时候不要急着 push

以下情况先停一下：

- `git status` 里出现你不认识的文件。
- 改动里包含密码、密钥、Token。
- 还没跑基本检查。
- 提交信息还写着 `update`。
- 不确定当前分支是不是目标分支。

先看清楚，再推送。GitHub 是协作平台，推上去的内容别人可能马上看到。
