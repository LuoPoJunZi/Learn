# 4.3 如何提一个高质量 Issue 或 Bug 报告

Issue 是 GitHub 上用来沟通问题、需求和任务的地方。它不只是“报错窗口”，也是团队协作的记录板。

## Issue 可以用来做什么

- 报告 Bug
- 提出新功能建议
- 记录待办任务
- 讨论设计方案
- 追踪文档改进

## 一个差的 Issue

```text
运行不了，快修。
```

这种描述太少，维护者不知道：

- 你做了什么
- 你期望发生什么
- 实际发生了什么
- 环境是什么
- 有没有报错信息

## 一个好的 Bug 报告

```markdown
## 问题描述

按照安装教程执行 git push 时失败。

## 复现步骤

1. 按照 1.3 配置 SSH 密钥
2. 克隆仓库
3. 修改 README.md
4. 执行 git push

## 期望结果

代码成功推送到 GitHub。

## 实际结果

终端提示 Permission denied。

## 环境信息

- 系统：Windows 11
- Git 版本：2.45.0
- 使用方式：PowerShell

## 补充信息

已确认 GitHub 里添加了公钥，但不确定是否是当前电脑生成的公钥。
```

## 提新功能建议

可以这样写：

```markdown
## 需求说明

希望增加一篇“如何撤销错误提交”的教程。

## 使用场景

新手经常误提交临时文件，需要知道如何安全撤销。

## 期望内容

- 撤销工作区修改
- 取消暂存
- 修改最近一次提交信息
- 不建议新手使用的危险命令
```

## Issue 标题怎么写

不推荐：

```text
救命
有问题
打不开
```

推荐：

```text
docs: SSH test command lacks Windows explanation
bug: git push fails with Permission denied
feature: add guide for undoing commits
```

## 提 Issue 前先做什么

- 搜索是否已有类似 Issue
- 确认自己使用的是最新文档
- 尽量提供截图、报错文本或命令输出
- 把敏感信息打码，例如 Token、邮箱、服务器 IP

## 维护者如何处理 Issue

常见处理方式：

- 添加 label，比如 `bug`、`docs`、`question`
- 请求更多信息
- 指派负责人
- 关联 Pull Request
- 问题解决后关闭 Issue

## 团队协作礼仪

- 描述事实，不要发泄情绪
- 不催促开源维护者
- 贴报错信息时注意脱敏
- 解决后补充结果，方便后来的人参考

一个好的 Issue，本质上是在帮别人快速进入现场。
