# 4.2 告别只会写 update：Commit 提交规范

Commit 是项目历史里的一个节点。好的提交信息能让别人快速理解：这次改了什么，为什么改。

## 为什么不要只写 update

下面这些提交信息几乎没有信息量：

```text
update
fix
改了
提交一下
```

过几天再看，你很难知道它们到底改了什么。

## 推荐格式

```text
类型: 简短说明
```

示例：

```text
docs: add GitHub setup guide
fix: correct SSH command
feature: add issue template
style: format markdown headings
```

## 常用类型

- `feature`: 新功能
- `fix`: 修复问题
- `docs`: 文档修改
- `style`: 格式调整，不改变含义
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建、配置、维护类修改

## 好的提交信息长什么样

推荐：

```text
docs: add SSH key setup guide
fix: correct remote repository URL example
chore: update gitignore for log files
```

不推荐：

```text
docs
final
111
改好了
```

## 一次提交应该多大

一次提交最好只表达一件事。

比如你同时做了三件事：

- 修错别字
- 增加 SSH 教程
- 修改 README 导航

可以拆成三次提交：

```bash
git commit -m "docs: fix typos in git basics"
git commit -m "docs: add SSH key setup guide"
git commit -m "docs: update README navigation"
```

这样以后查历史、回退修改都更容易。

## 提交前检查

提交前建议运行：

```bash
git status
git diff --staged
```

确认：

- 没有提交无关文件
- 没有提交密码或临时文件
- 提交内容和提交信息一致
- 文件能正常打开或运行

## 中文还是英文

团队统一即可。

如果项目主要面向中文读者，可以写：

```text
docs: 增加 SSH 密钥配置教程
fix: 修正远程仓库地址示例
```

如果项目开源给全球用户，建议使用英文。

## 常见建议

- 用现在时描述修改，例如 `add`、`fix`、`update`
- 第一行尽量简短
- 不要把多个无关修改塞进一次提交
- 不确定类型时，文档类就用 `docs`
