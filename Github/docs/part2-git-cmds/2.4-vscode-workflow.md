# 2.4 用 VS Code 丝滑提交

Git 不一定非要在黑框框里用。VS Code 自带 Git 图形界面，适合新手理解“修改、暂存、提交、推送”的流程。

## 从 GitHub 克隆到本地

如果项目已经在 GitHub 上，第一步通常是把它克隆到本地电脑。

1. 在 VS Code 中点击 `Terminal` -> `New Terminal`。
2. 在终端中输入：

```bash
git clone https://github.com/用户名/仓库名.git
```

如果你已经配置了 SSH，也可以使用：

```bash
git clone git@github.com:用户名/仓库名.git
```

克隆完成后，当前目录会出现一个新的项目文件夹。

非常重要：接下来要在 VS Code 里点击 `File` -> `Open Folder`，重新打开刚刚克隆下来的项目根目录。只有打开的是项目根目录，VS Code 才能正确识别 Git 仓库、显示分支和修改记录。

## 打开项目

1. 打开 VS Code。
2. 点击 `File` -> `Open Folder`。
3. 选择你的项目文件夹。

如果这个文件夹已经是 Git 仓库，左侧源代码管理图标会显示修改数量。

## 认识源代码管理面板

左侧有一个像分叉线的图标，叫 Source Control。

在这里你可以看到：

- 修改了哪些文件
- 新增了哪些文件
- 删除了哪些文件
- 哪些文件已暂存
- 提交信息输入框

## 查看文件差异

点击一个被修改的文件，VS Code 会展示修改前后的对比。

常见颜色含义：

- 绿色：新增内容
- 红色：删除内容
- 蓝色或高亮：修改内容

提交前先看差异，是非常好的习惯。

## 暂存修改

在 Source Control 面板里：

- 点击文件旁边的 `+`，暂存单个文件。
- 点击 Changes 旁边的 `+`，暂存所有修改。

对应命令是：

```bash
git add 文件名
git add .
```

## 提交修改

在输入框里写提交信息，例如：

```text
docs: add git setup guide
```

然后点击 `Commit`。

对应命令是：

```bash
git commit -m "docs: add git setup guide"
```

## 推送和拉取

提交后，VS Code 通常会在底部状态栏或 Source Control 面板里显示同步按钮。

- Push：把本地提交上传到 GitHub。
- Pull：把 GitHub 上的新提交拉到本地。
- Sync：通常等于先拉取再推送。

团队协作时，建议先 Pull，再 Push。

## GitHub.dev 和本地 VS Code 的区别

GitHub 网页上按 `.` 可以打开 GitHub.dev，它很适合临时改文档或修错别字。但如果要认真开发，建议使用本地 VS Code。

| 功能 | 本地 VS Code | GitHub.dev |
| :--- | :--- | :--- |
| 终端 | 有，可以运行 Git、脚本和测试命令 | 通常没有完整终端 |
| 插件 | 支持完整插件生态 | 只支持部分 Web 插件 |
| 离线工作 | 可以离线修改和提交，本地保存历史 | 依赖网络 |
| 搜索与批量修改 | 速度快，适合大项目 | 适合轻量编辑 |
| 适用场景 | 深度开发、调试、批量修改 | 临时阅读、轻微修改 |

简单理解：GitHub.dev 像轻量编辑器，本地 VS Code 才是完整开发环境。

## 创建分支

点击 VS Code 左下角的分支名，例如 `main`，可以：

- 创建新分支
- 切换分支
- 发布分支到 GitHub

新功能建议在新分支上做，不要直接改 `main`。

## 解决冲突

如果 VS Code 检测到冲突，会在文件里显示几个操作按钮：

- Accept Current Change：保留当前分支内容
- Accept Incoming Change：保留传入分支内容
- Accept Both Changes：两个都保留
- Compare Changes：对比查看

解决后，需要重新暂存并提交。

## 推荐扩展

### GitLens

GitLens 可以显示每一行是谁修改的、什么时候修改的、对应哪次提交。

适合排查问题：

- 这段代码谁改的？
- 为什么这里会变成这样？
- 这个文件最近有哪些修改？

### GitHub Pull Requests

这个扩展可以在 VS Code 里查看和处理 Pull Request。

适合团队协作：

- 查看 PR 评论
- 本地检出 PR 分支
- 参与代码审查

### ShellCheck

如果你的项目里有 Bash 或 Shell 脚本，建议安装 ShellCheck 相关扩展。它可以在保存前提示常见脚本错误，例如语法错误、变量引用问题、危险写法等。

这类检查能帮你在推送到 GitHub 前发现问题，减少“提交后才发现脚本跑不起来”的情况。

## 常见卡住场景

### 为什么修改了文件却不能提交

在 VS Code 的 Source Control 面板中，文件通常会先出现在 `Changes` 列表里。它只是表示“这个文件改过”，还没有进入本次提交。

你需要先点击文件右侧的 `+`，把它放入 `Staged Changes`。这一步对应命令：

```bash
git add 文件名
```

然后再填写提交信息并点击 `Commit`。

记住这条顺序：

```text
修改文件 -> 点击 + 暂存 -> 写提交信息 -> Commit -> Push 或 Sync
```

### Sync 或 Push 时提示无法连接 GitHub

如果你看到类似报错：

```text
Failed to connect to github.com port 443
Couldn't connect to server
```

通常说明本地电脑连接 GitHub 超时。注意：如果你已经成功点击 `Commit`，说明修改已经保存在本地仓库里，只是还没有推送到 GitHub。

可以先检查网络是否正常，再重试 `Push` 或 `Sync`。

如果你的网络环境需要走本地代理，可以让 Git 使用本机代理端口。常见端口示例：

- Clash 常见端口：`7890`
- v2rayN 常见端口：`10808`

请以你自己的代理软件实际端口为准。假设端口是 `7890`，可以执行：

```bash
git config --global http.proxy http://127.0.0.1:7890
git config --global https.proxy http://127.0.0.1:7890
```

设置后重新点击 VS Code 的 `Sync Changes`，或在终端执行：

```bash
git push
```

如果以后不需要代理，或端口变化了，可以取消配置：

```bash
git config --global --unset http.proxy
git config --global --unset https.proxy
```

查看当前 Git 代理配置：

```bash
git config --global --get http.proxy
git config --global --get https.proxy
```

如果输出为空，说明没有设置全局代理。

## 新手工作流

推荐你按这个顺序操作：

1. 打开 Source Control。
2. 查看修改文件。
3. 点击文件看差异。
4. 暂存确认要提交的文件。
5. 写清楚提交信息。
6. Commit。
7. Pull。
8. Push。

VS Code 让 Git 变得可视化，但背后的逻辑仍然是 Git。理解命令含义后，再用图形界面会更稳。
