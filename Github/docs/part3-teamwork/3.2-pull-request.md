# 3.2 如何发起和合并 Pull Request

Pull Request 常简称 PR。它的意思是：我完成了一些修改，请团队检查一下，如果没问题，就把它合并进主分支。

## PR 解决什么问题

PR 让团队可以在合并代码前做这些事：

- 查看这次改了哪些文件
- 讨论实现方式
- 发现潜在问题
- 跑自动化测试
- 留下完整的协作记录

## 发起 PR 前要做什么

确保你的分支已经推送到 GitHub：

```bash
git push -u origin feature/my-work
```

确保提交信息清楚，不要堆很多无意义提交。

如果主分支有更新，建议先同步：

```bash
git switch main
git pull
git switch feature/my-work
git merge main
```

如果出现冲突，先解决冲突再推送。

## 在 GitHub 上创建 PR

1. 打开你的 GitHub 仓库。
2. 页面通常会提示 `Compare & pull request`。
3. 点击后确认：
   - base 分支通常是 `main`
   - compare 分支是你的功能分支
4. 填写 PR 标题和说明。
5. 点击 `Create pull request`。

## PR 标题怎么写

标题要让别人一眼知道你做了什么。

不推荐：

```text
update
fix
修改了一些东西
```

推荐：

```text
docs: add GitHub setup guide
fix: correct SSH test command
feature: add issue template
```

## PR 描述可以写什么

一个清楚的 PR 描述通常包含：

```markdown
## 修改内容

- 增加 Git 安装教程
- 补充 SSH 密钥配置说明

## 为什么修改

新手在第一次配置环境时容易卡住，需要更完整的步骤。

## 自测

- 已检查 Markdown 链接
- 已按 Windows 流程测试命令
```

## 代码审查时怎么回应

别人提评论不是在否定你，而是在帮项目变好。

建议做法：

- 看不懂就问清楚
- 同意就修改并回复
- 不同意时说明理由
- 修改后重新 push，PR 会自动更新

## 合并 PR

当审查通过、测试通过后，可以点击 `Merge pull request`。

常见合并方式：

- Merge commit：保留完整分支历史。
- Squash and merge：把多个提交压成一个提交。
- Rebase and merge：让历史更线性。

新手团队可以先使用 GitHub 默认的合并方式，等熟悉后再统一规范。

## 合并后做什么

切回主分支并更新：

```bash
git switch main
git pull
```

删除已经合并的功能分支：

```bash
git branch -d feature/my-work
```

## 高质量 PR 检查清单

- 标题能说明这次修改的目的
- 描述里写了改动内容
- 没有提交无关文件
- 没有上传密码、日志、临时文件
- 自己先看过 diff
- 能解释为什么这样改
