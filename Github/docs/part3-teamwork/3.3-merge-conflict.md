# 3.3 多人修改同一个文件导致冲突怎么办？

冲突并不可怕。它只是 Git 在告诉你：同一个地方被不同人改了，我不知道该保留哪一份，需要你来决定。

## 冲突通常什么时候出现

常见场景：

- 你和别人修改了同一个文件的同一行
- 你删除了一个文件，别人修改了这个文件
- 你基于旧版本开发，主分支已经变化很多

常见命令中可能出现冲突：

```bash
git pull
git merge main
git rebase main
```

## 冲突长什么样

文件里可能出现：

```text
<<<<<<< HEAD
这是你当前分支的内容
=======
这是别人传入的内容
>>>>>>> main
```

这些标记不是正常内容，需要你处理掉。

## 如何解决冲突

解决冲突的本质是做选择：

- 保留你的内容
- 保留别人的内容
- 两边内容都保留
- 手动改成一个新的版本

处理后，文件里不能再有：

```text
<<<<<<<
=======
>>>>>>>
```

## 命令行解决流程

先查看冲突文件：

```bash
git status
```

打开文件，手动编辑冲突部分。

解决后暂存：

```bash
git add 冲突文件名
```

如果是 merge 冲突，继续提交：

```bash
git commit
```

如果是 rebase 冲突，继续 rebase：

```bash
git rebase --continue
```

## VS Code 解决冲突

VS Code 会在冲突文件上显示操作按钮：

- Accept Current Change：保留当前分支
- Accept Incoming Change：保留传入内容
- Accept Both Changes：两边都保留
- Compare Changes：对比查看

选择后仍然要检查最终文件内容是否正确。

## 冲突处理示例

冲突内容：

```text
<<<<<<< HEAD
欢迎阅读 Git 教程
=======
欢迎阅读 GitHub 新手教程
>>>>>>> main
```

你可以改成：

```text
欢迎阅读 GitHub 新手教程
```

或者：

```text
欢迎阅读 Git 与 GitHub 新手教程
```

最后删除冲突标记，保存文件。

## 如何减少冲突

- 开始工作前先 `git pull`
- 小步提交，不要一次改太多
- 修改公共文件前和队友沟通
- 不要长时间让分支落后主分支
- 及时发 PR，及时合并

## 不建议新手随便使用的命令

```bash
git reset --hard
git push --force
```

这些命令可能丢失修改或覆盖远程历史。没有把握时，先备份项目文件夹，再请有经验的人一起看。
