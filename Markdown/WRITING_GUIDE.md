# Markdown 写作进阶指南

这篇文档不重复基础语法，而是讲“如何把 Markdown 写成真正好读的教程、README 和博客”。新手学会语法后，最需要提升的是结构感。

## 好文档的判断标准

一篇好文档通常满足：

- 第一屏能看出主题和用途。
- 标题层级清楚。
- 代码块前后有解释。
- 链接有上下文，不是裸贴。
- 危险操作有提醒。
- 读者知道下一步去哪里。

## README 写作模板

推荐结构：

```markdown
# 项目名称

一句话说明这个项目做什么。

## 适合谁

- 用户 A
- 用户 B

## 功能或目录

| 入口 | 说明 |
| :--- | :--- |
| docs | 教程文档 |

## 如何使用

步骤说明。

## 常见问题

### 问题

回答。

## 许可证或来源
```

README 的目标不是炫技，而是让第一次打开的人不迷路。

## 教程文档写法

教程建议按“目标、前置条件、步骤、验证、常见问题、下一步”组织。

示例结构：

```markdown
# 如何配置 SSH 密钥

## 适合人群

## 学习目标

## 前置条件

## 操作步骤

## 如何验证成功

## 常见问题

## 下一步
```

新手教程一定要写“如何验证成功”。没有验证步骤，读者不知道自己做对没有。

## 博客文章写法

博客更适合记录过程和思考，不一定像教程那么规整。

推荐结构：

```text
我遇到了什么问题
-> 我尝试了哪些方法
-> 哪一步真正解决了
-> 关键命令或代码
-> 我学到了什么
```

如果是踩坑文章，保留失败路径很有价值。别人很可能也会踩到同一个坑。

## 表格怎么用才清楚

表格适合比较短信息：

| 适合表格 | 不适合表格 |
| :--- | :--- |
| 命令和作用 | 很长的解释段落 |
| 文件和说明 | 多层嵌套内容 |
| 参数和默认值 | 大段代码 |

如果表格在手机上太宽，改成列表会更好。

## 代码块怎么写

代码块前面说明“这段代码做什么”，后面说明“预期看到什么”。

```powershell
git status
```

预期结果：你会看到哪些文件被修改、哪些文件已暂存。

不要只贴命令，不解释用途。新手最怕的是“照抄能跑，但不知道为什么”。

## Mermaid 使用建议

Mermaid 适合说明流程，不适合塞太多细节。

推荐：

````markdown
```mermaid
flowchart LR
    A["修改文件"] --> B["提交"]
    B --> C["推送"]
```
````

节点文字较长时，用引号包起来，减少渲染问题。

## 平台迁移注意事项

从 GitHub 迁移到博客时，重点检查：

- Mermaid 是否支持。
- 数学公式是否支持。
- 图片路径是否仍然有效。
- 任务列表是否正常显示。
- HTML 标签是否被过滤。

通用文档优先使用基础 Markdown，平台专属语法要写说明。

## 下一步

写完文档后，运行仓库的文档检查脚本：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\check-docs.ps1
```

然后从读者角度打开 README，问自己：第一次来的人能不能在 30 秒内知道下一步点哪里？
