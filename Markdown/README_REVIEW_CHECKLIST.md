# README 审查清单

这份清单用于检查一篇 README 是否对新手友好。写完文档后，从读者角度过一遍，常常能发现很多“作者以为很明显，但读者完全不知道”的地方。

## 第一屏

检查：

- 标题是否清楚。
- 开头是否一句话说明项目用途。
- 读者能否在 30 秒内知道下一步看哪里。
- 是否说明适合谁阅读或使用。

如果第一屏只堆徽章、口号或长背景，新手会很快迷路。

## 目录和入口

检查：

- 重要目录是否有说明。
- 链接是否能打开。
- 路径中有空格时是否使用尖括号。
- 是否有“推荐阅读顺序”。
- 新增文件是否已经挂到 README。

本仓库可以运行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\check-docs.ps1
```

## 安装和运行

检查：

- 是否写明需要的软件版本。
- 是否说明命令在哪个目录运行。
- 是否区分 Windows、Linux、macOS。
- 是否写明输入文件和输出结果。
- 是否说明如何验证成功。

教程里最容易漏掉的是“如何验证成功”。没有验证，读者只能猜。

## 命令和代码块

检查：

- 代码块是否标注语言。
- 命令前是否说明用途。
- 命令后是否说明预期结果。
- 危险命令是否有提醒。
- 不要把密码、Token 写进示例。

示例：

```powershell
git status
```

用途：查看当前仓库是否有未提交修改。

## 常见问题

检查：

- 是否列出新手最可能遇到的 3 到 5 个问题。
- 回答是否具体到可以执行。
- 是否链接到更详细的排查文档。
- 是否说明不要做哪些危险操作。

FAQ 不需要覆盖所有情况，先覆盖最容易卡住的地方。

## 来源和许可证

检查：

- 外部资源是否标注来源。
- 复制或改写内容是否符合许可证。
- 是否说明图片、模板、代码的来源。
- 是否保留必要署名。

如果不确定许可证，不要直接搬运源码或素材。

## 最后快速检查

提交前问自己：

- 这篇 README 是给谁看的？
- 新手能不能照着完成第一步？
- 所有链接是否能打开？
- 有没有泄露敏感信息？
- 有没有与实际目录不一致的描述？

## 下一步

如果 README 已经能通过这份清单，可以继续看 [Markdown 写作进阶指南](WRITING_GUIDE.md)，优化教程结构和示例表达。
